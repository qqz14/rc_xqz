##  需求分析

根据需求描述，这是一个**可靠的外部 HTTP 通知投递服务**，核心诉求是：

- 接收业务系统提交的通知请求
- 可靠地投递到不同供应商的 HTTP(S) API
- 业务系统不关心返回值，只关心"能送达"

## ️ 初步模块方案（请确认）

根据需求分析，本系统拟拆分为以下核心模块：

| 模块编号 | 模块名称 | 职责说明 | 计划使用的图表类型 |
|---------|---------|---------|------------------|
| M1 | **接入层（Submission API）** | 接收业务系统提交的通知任务，做参数校验、持久化入队 | 时序图 |
| M2 | **任务持久化层（Task Store）** | 将通知任务写入持久化存储（DB/队列），保证不丢失 | ER 图 |
| M3 | **投递引擎（Dispatcher）** | 从队列消费任务，向目标地址发起 HTTP 请求 | 流程图 + 时序图 |
| M4 | **重试调度器（Retry Scheduler）** | 对失败任务按指数退避策略重新调度 | 状态图 + 流程图 |
| M5 | **死信处理（Dead Letter Handler）** | 超过最大重试次数后，将任务转入死信区，支持告警和人工干预 | 状态图 |

**数据存储规划**：

| 表名/存储介质 | 操作类型 | 用途说明 |
|-------------|---------|---------|
| `notification_tasks` | 新增 | 存储所有通知任务及其状态 |
| `notification_logs` | 新增 | 记录每次投递尝试的请求/响应详情 |
| 消息队列（Redis List 或 MQ） | 使用 | 作为投递任务的缓冲与调度介质 |

**核心技术决策建议**：

1. **投递语义**：选用 **At-Least-Once（至少一次）**，通过幂等性要求由调用方保障
2. **重试策略**：指数退避（1s → 2s → 4s → ... → 上限），最大重试次数可配置（建议默认 5 次）
3. **队列选型**：优先使用 **Redis（ZSET 延迟队列）** 作为 MVP 方案，无需引入重型 MQ；若流量增长可迁移至 RabbitMQ/Kafka
4. **持久化**：任务先写 DB 再入队，保证崩溃恢复能力
